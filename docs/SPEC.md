# 커플 가계부 앱 — 설계 문서 (SPEC)

> 둘이서 카드 사용내역을 함께 보며 정리하는 사적인 가계부 앱
> 최종 수정: 2026-06-04

---

## 1. 목적 & 범위

- **목적**: 본인 + 여자친구 두 명만 쓰는 사적인 가계부. 카드 사용내역을 같이 보며 정리하는 것이 핵심.
- **핵심 가치**: *지출 한눈에 모아보기* (이번 달 총지출 + 카테고리 + 최근 내역).
- **사용자**: 딱 2명. 회원가입/멀티테넌시/앱스토어 배포 불필요.
- **MVP 원칙**: 처음부터 단순하게. 카드 1개, 이메일 1개. 확장은 나중에.

### 명시적으로 안 하는 것 (MVP 제외)
- 여러 카드/여러 이메일 동시 수집
- 공동지출 vs 개인지출 정산 (카드가 2개 이상 들어올 때 추가)
- 예산/절약 알림
- 네이티브 앱 (PWA로 충분)

## 2. 자동수집 파이프라인 (앱의 심장)

### 결정 사항
- **수집원**: 현대카드 "이용내역 이메일 알림" 1개
- **이메일**: 개인 Gmail 1개 (전용 계정 아님)
- **방식**: Gmail API로 서버가 폴링 → 파싱 → DB 저장
- **아이폰 제약**: iOS는 앱이 SMS를 못 읽음 → 기기가 아니라 **서버에서 이메일로** 수집

### 흐름
```
현대카드 결제
   → 현대카드가 "이용내역 이메일" 발송
   → 개인 Gmail 수신 (필터로 `가계부` 라벨 자동 부착)
   → 오라클 서버가 Gmail API로 해당 라벨 메일만 폴링
   → 메일 파싱 (가맹점/금액/일시/카드)
   → DB에 거래 저장 (중복 방지)
   → 웹앱에 표시
```

### 구현 메모
- Gmail 필터: 현대카드 발신주소 → `가계부` 라벨 자동 부착. 서버는 `label:가계부 is:unread` 류로 조회 후 처리, 처리 완료 메일은 읽음/별도 라벨 처리하여 중복 방지.
- OAuth 토큰은 서버에만 안전 보관. 권한 범위는 가능한 한 좁게 (readonly + 라벨 조회).
- 중복 방지 키: Gmail messageId + 파싱된 (가맹점, 금액, 승인일시) 조합.
- **선결 과제**: 현대카드 이용내역 메일의 실제 HTML/텍스트 양식 확보 → 파싱 규칙 작성 (다음 작업).

## 3. 아키텍처

```
인터넷
  │
  ▼
[Cloudflare]  ← 서버 IP 숨김, 봇/스캔 차단, Cloudflare Access로 둘 이메일만 허용
  │
  ▼
[오라클 클라우드 Always Free VM]
  ├─ 이메일 폴링 + 파싱 워커  (Gmail API)
  ├─ DB (PostgreSQL 또는 SQLite)
  └─ 웹앱 (PWA, 백엔드+프론트)
  │
  ▼
[너 아이폰]  [여친 아이폰]  ← 홈화면에 추가한 PWA
```

### 서버: 오라클 클라우드 Always Free
- ARM Ampere VM (최대 4 vCPU / 24GB RAM) — 평생 무료, 둘이 쓰기 충분
- 이메일 24시간 폴링하므로 미사용 회수 정책에 안 걸림
- 고정 공인 IP 무료, DB도 같은 서버에

### 기술 스택 (제안, 확정 전)
- **풀스택**: Next.js (백엔드+프론트 한 몸) 또는 Node + 경량 프론트
- **DB**: PostgreSQL (또는 단순하게 SQLite)
- **이메일**: Gmail API (무료, OAuth 1회 인증)
- **배포**: 오라클 서버에 Docker

## 4. 보안 / 접근 제한

카드 내역은 민감정보 → "주소를 아는 사람만"은 금지. 여러 층으로 방어:

1. **Cloudflare 앞단**: 서버 진짜 IP 숨김, 스캔/봇 차단
2. **Cloudflare Access** (무료): 두 분 이메일 2개만 허용 — 나머지는 페이지 접근 불가
3. **앱 로그인**: 둘만 아는 인증 (이중 안전장치), Passkey/기기 등록 고려
4. **서버 방화벽**: 80/443만 열고, Cloudflare에서 오는 트래픽만 허용

> 향후 더 강하게: Tailscale 등 VPN으로 우리 기기끼리만 연결된 사설망에 서버 배치.

## 5. 화면 설계

### 5.1 대시보드 (첫 화면)
```
┌─────────────────────────────┐
│  2026년 6월          ◀ ▶     │  월 이동
├─────────────────────────────┤
│     이번 달 총 지출           │
│      1,234,500원            │  제일 크게
│   지난달보다 ▲ 12%          │  비교
├─────────────────────────────┤
│  카테고리별                  │
│  🍚 식비   480,000  ▓▓▓▓    │
│  🚗 교통   120,000  ▓▓      │  막대로 비중
│  🛒 쇼핑   300,000  ▓▓▓     │
│  … 더보기                    │
├─────────────────────────────┤
│  최근 내역            전체▶  │
│  오늘                        │
│   스타벅스 강남    -5,800   │
│   GS25           -12,300   │
│  어제                        │
│   쿠팡           -34,000   │  자동수집분이 쌓임
├─────────────────────────────┤
│        [ + 직접 입력 ]       │  현금/누락 보충
└─────────────────────────────┘
```
- 맨 위 = 이번 달 총액 (앱 열자마자 즉답)
- 카테고리 막대 = 어디에 많이 쓰는지 (대화의 시작점)
- 최근 내역 = 자동수집된 결제가 시간순으로
- + 직접 입력 = 현금/누락분 보충

### 5.2 거래 상세 (같이 정리하는 곳)
```
스타벅스 강남
  -5,800원   2026.06.04 14:30
  카테고리  [ ☕ 카페  ▾ ]
  메모      [ 여친이랑 ___ ]
  ☑ 자동수집됨 (현대카드)
```
- 카테고리 변경, 메모 추가
- 둘 다 같은 데이터 → 한 명이 정리하면 상대 화면에도 반영 (실시간 공유)

## 6. 데이터 모델 (초안)

### transactions (거래)
| 필드 | 타입 | 설명 |
|------|------|------|
| id | PK | |
| merchant | text | 가맹점명 |
| amount | int | 금액(원), 지출 음수/양수 규칙은 구현 시 확정 |
| occurred_at | datetime | 결제 일시 |
| category | text/FK | 카테고리 |
| memo | text | 메모 (둘이 정리) |
| source | enum | `email_auto` / `manual` |
| card | text | 카드명 (현재 현대카드) |
| gmail_message_id | text | 중복 방지용 (자동수집분) |
| created_by | text | 입력자 (수기 입력 시) |
| created_at | datetime | |

### categories (카테고리)
- 식비, 교통, 쇼핑, 카페, 등 — 우리 둘 소비 패턴에 맞게 커스텀

## 7. 확정 사항 요약

| 영역 | 결정 |
|------|------|
| 목적 | 둘이 쓰는 사적 가계부, 지출 한눈에 모아보기 |
| 자동수집 | 현대카드 이용내역 이메일 → 개인 Gmail → 서버 파싱 |
| 수기 입력 | 보조 (현금·누락분) |
| 서버 | 오라클 Always Free + DB + 웹앱(PWA) |
| 접근제한 | Cloudflare Access(둘만) + 앱 로그인 |
| 첫 화면 | 이번 달 총지출 + 카테고리 + 최근 내역 + 직접입력 |
| 카드/이메일 | MVP는 각 1개. 확장 시 공동/개인 구분·정산 추가 |
